# FerroTone: System Architecture & Execution Plan

This document provides a detailed breakdown of the architectural boundaries, internal communications, data formats, 
and development phases for FerroTone — an open-source, cross-platform vocal training desktop application.

## 1. Architectural Boundaries & Data Flow

FerroTone splits engineering tasks across three layers based on performance characteristics, 
ease of development, and existing ecosystem capabilities:

```
┌────────────────────────────────────────────────────────┐
│                   FRONTEND (UI LAYER)                  │
│                TypeScript / React / Vite               │
│   • HTML5 Canvas Render Loop (60 FPS Game Loop)        │
│   • State Management (Current Time, Active Track)      │
└───────────────────────────┬────────────────────────────┘
                            │ Tauri IPC (Commands & Events)
┌───────────────────────────▼────────────────────────────┐
│                    TAURI CORE (RUST)                   │
│   • Thread 1: Main App Event loop & Tauri Context      │
│   • Thread 2: Mic Capture Engine (`cpal` Buffer)      │
│   • Thread 3: Pitch Extraction (`pitch-detection`)     │
└───────────────────────────┬────────────────────────────┘
                            │ Local Localhost Socket / IPC
┌───────────────────────────▼────────────────────────────┐
│                  AI ENGINE (PYTHON SIDECAR)            │
│   • Subprocess / Socket Listener                       │
│   • Audio Demixing Execution (Spleeter / Demucs)        │
└────────────────────────────────────────────────────────┘
```

### A. The Front-End (UI Layer)

**Role:** User dashboard, audio playback triggers, rendering the gamified scoring view.

**Performance Requirement:** Stable 60 FPS rendering when painting the moving pitch canvas.

**Strategy:** Avoid framework re-renders for streaming audio data. 
Instead, establish a direct reference to an HTML5 Canvas node and let a local JavaScript array buffer draw 
incoming pitch data inside a `requestAnimationFrame` loop.

### B. The Core Engine (Rust Layer)

**Role:** Audio device configuration and low-latency DSP.

**Threading Model:**

- **Main Thread:** Manages Tauri window states and UI Command routing.
- **Audio I/O Thread:** Managed by `cpal`. Listens directly to the hardware input buffer.
- **DSP Worker Thread:** Ingests raw audio frames from the I/O thread, applies downsampling or windowing, runs pitch calculation algorithms (YIN/MPM), and throws the derived pitch up to the Frontend via Tauri events.

### C. The AI Engine (Python Sidecar Layer)

**Role:** Asynchronous, computation-heavy file processing.

**Strategy:** Kept completely separate to avoid blocking the user experience. 
Invoked via a Tauri background shell command. 
It processes an audio file, stores the results in a local user configuration folder, and returns a structural JSON metadata schema.

## 2. Core Data Schemas

To ensure clean interoperability between Rust, Python, and TypeScript, data structures must follow strict, well-defined formats.

### Target Track Profile Blueprint (Generated by Python/Rust)

When a song is split, the vocal stem is analyzed once to generate a timeline profile of what the frequencies should be. 
This maps directly to a JSON payload loaded by the UI before playback starts:

```json
{
  "track_metadata": {
    "title": "Bohemian Rhapsody",
    "artist": "Queen",
    "duration_seconds": 354.2
  },
  "pitch_timeline": [
    { "time_ms": 1200, "hz": 220.0, "note": "A3", "confidence": 0.94 },
    { "time_ms": 1233, "hz": 220.0, "note": "A3", "confidence": 0.95 },
    { "time_ms": 1266, "hz": 223.4, "note": "A3", "confidence": 0.89 }
  ]
}
```

### Live Sing-Along Frame (Emitted by Rust at ~30-63 Hz)

As the user sings, the Rust core fires asynchronous events to the frontend containing data structurally identical to this payload:

```json
{
  "timestamp_ms": 1235,
  "live_hz": 218.2,
  "live_note": "A3",
  "rms_volume": 0.045
}
```

## 3. Step-by-Step Implementation Roadmap

We will divide production into four distinct, highly isolated development phases to guarantee immediate code feedback loops.

### Phase 1: Native Core & Real-Time Pitch (Weeks 1-2)

**Goal:** Build a stable, ultra-low-latency pitch tracking terminal loop.

**Tasks:**
- Configure `cpal` to latch to the standard OS default input device.
- Stream input buffers safely into the `pitch-detection` crate utilizing the YIN algorithm framework.
- Expose a Tauri command loop that outputs the parsed fundamental frequency (f₀) to the console.

**Success Metric:** Sing into your mic; your console cleanly outputs fluctuating Hz numbers matching your voice in real time.

### Phase 2: The Graphics & Game Loop (Weeks 3-4)

**Goal:** Create a visual scrolling pitch matching matrix.

**Tasks:**
- Design an HTML5 Canvas Component inside React.
- Write a mathematical transformer to convert raw Hertz into standard vertical screen space (Y-axis pixels) based on the musical logarithmic scale.
- Mock a stationary track line, stream simulated user input data from Rust, and draw scrolling target vectors crossing paths with the user's current pitch cursor.

**Success Metric:** A fluid, 60 FPS visualization scrolling across the screen where singing higher or lower guides a tracking dot up and down relative to a target horizontal line.

### Phase 3: System Loopback & The AI Sidecar (Weeks 5-7)

**Goal:** Ingest exterior audio tracks and slice them cleanly into individual stems.

**Tasks:**
- Integrate PyAudioWPatch on Windows to facilitate speaker capturing.
- Construct the local isolated Python processing scripts containing Spleeter/Demucs.
- Build a Tauri Sidecar harness wrapper allowing the application to pass audio path strings directly into the executable script.
- Process output stems (`vocals.wav` and `accompaniment.wav`), saving them directly to the native client cache.

**Success Metric:** Selecting a local track cleanly outputs a dedicated vocal track file alongside an instrumental file inside the application workspace directory.

### Phase 4: Gamification, Calibration, & Ear Training (Weeks 8-10)

**Goal:** Fine-tune latency tolerances, score accuracy, and roll out interactive training UI views.

**Tasks:**
- Implement a latency calibration widget to correct physical hardware delay differences between the speaker playback and microphone inputs.
- Write a point scoring math calculation component that grades user accuracy against target intervals (|Target Hz - User Hz|).
- Implement dedicated training workflows (e.g., scale steps, octave leaps, static interval matching).

**Success Metric:** Users can interact with menu screens, run short ear-training note scales, and receive a performance accuracy score at the completion of a track.

## 4. Immediate Engineering Priorities

To begin programming without getting overwhelmed, focus on completing these items in order:

1. **GitHub Sanity Check:** Ensure the initial repository push compiles cleanly from your workstation via `npm run tauri dev`.
2. **Audio Stack Configuration:** Confirm your updated `src-tauri/src/main.rs` file successfully captures your microphone using `cpal` and accurately paints the console volume meter.
3. **Crate Testing:** Add the `pitch-detection` crate to your dependencies and replace the basic volume logic with the YIN algorithm calculation block.
